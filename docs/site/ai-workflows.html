Guide 03

Give the model objects, not stdout.

Tarn's wedge is not "YAML tests exist." It is that failures come back as structured data with a stable taxonomy that an agent can reason over directly.

Use JSON intentionally

$ tarn run --format json --json-mode compact
$ tarn run --format human --format json=reports/run.json --format curl=reports/failures.sh
  • Verbose JSON: richer assertion detail for forensics.
  • Compact JSON: slimmer payload for LLM and automation loops.
  • curl export: replays failed or full-suite requests without re-reading YAML.

Fields that matter in the retry loop

  • failure_category: assertion, parse, connection, timeout, capture.
  • error_code: stable machine-readable subtype for routing and heuristics.
  • remediation_hints: lightweight next-step guidance in the JSON itself.
  • request and response: included on executed failures so the model sees what actually happened.

Recommended shell loop

  1. Generate or edit the Tarn YAML.
  2. Run tarn validate to catch parser and config issues.
  3. Run tarn run --format json --json-mode compact.
  4. Branch on failure_category and error_code before touching assertions blindly.
  5. Use curl export when you need to replay exact failures outside Tarn.

Streaming and scoped execution

For live UIs and incremental retry loops, tarn run --ndjson streams one JSON event per line to stdout: file_started, step_finished, test_finished, file_finished, done. Pair it with --select FILE[::TEST[::STEP]] (repeatable, ANDs with --tag) to run just the failing step, or --only-failed to replay the previous run's failures. Use --no-progress in CI to keep the stream clean.

Editor intelligence via tarn-lsp

tarn-lsp is a standalone LSP 3.17 language server, separate from the CLI. Install with cargo install tarn-lsp. When an agent authors .tarn.yaml inside an LSP-aware editor, it gets schema-aware diagnostics, hover with inline JSONPath evaluation against a recorded response, nested completion, document symbols, go-to-definition, references, rename, code lens (tarn.runTest / tarn.runStep), whole-document formatting, three code actions (extract env var, capture this field, scaffold assert from recorded response), and a quick-fix code action backed by the same tarn::fix_plan engine as tarn_fix_plan. It also exposes tarn.evaluateJsonpath via workspace/executeCommand, so an agent can ask the server to evaluate a JSONPath against the sidecar response without re-running the suite. Full spec at docs/TARN_LSP.md.

Claude Code plugin

The tarn-lsp Claude Code plugin wires the language server into Claude Code's plugin/LSP system.

$ /plugin marketplace add /absolute/path/to/editors/claude-code
$ /plugin install tarn-lsp@tarn-lsp --scope project
$ /reload-plugins

The plugin claims all .yaml and .yml files because .tarn.yaml is a compound extension. Install with --scope project in Tarn-focused repositories so it does not take over YAML files elsewhere.

Recorded response sidecar

Several LSP features (inline JSONPath hover, the "scaffold assert from recorded response" code action, and tarn.evaluateJsonpath) read from a sidecar layout next to the test file. When writing a tool that hands a recorded response to tarn-lsp, drop the JSON at <file>.tarn.yaml.last-run/<test-slug>/<step-slug>.response.json.

VS Code extension

The Tarn VS Code extension (publisher nazarkalytiuk, available on the VS Marketplace and Open VSX) gives agents a test-explorer surface: discovery, CodeLens (Run / Dry Run / Run step), live streaming via --ndjson, unified-diff failure peek, environment picker, status bar, trusted/untrusted workspace handling, and Remote Development across Dev Container, Codespaces, WSL, and Remote SSH. Off-by-default tarn.experimentalLspClient scaffolding is in place for future direct consumption of tarn-lsp. Architecture notes at docs/VSCODE_EXTENSION.md.

Redaction

Report-time redaction is configurable at DSL level. Tarn can redact sensitive headers, named env vars, and selected captures across JSON, human, JUnit, TAP, HTML, and curl outputs.

Why this matters

An agent should not need to infer whether a failure is DNS, TLS verification, unresolved interpolation, or a plain status mismatch from plain-text output. Tarn ships that distinction directly.
